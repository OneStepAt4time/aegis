# ADR-0006: SSO/OIDC Integration for Dashboard Authentication

**Status:** Accepted
**Date:** 2026-04-10
**Issue:** #1410

## Context

Aegis currently supports only API key authentication (SHA-256 hashed bearer tokens). The dashboard (`/dashboard/`) is served with no authentication. Enterprise deployments require integration with corporate identity providers (Okta, Azure AD, Google Workspace) for browser-facing dashboard access. The API key model must be retained for machine-to-machine (M2M) API access.

## Decision

Use `openid-client` as the OIDC Relying Party library for dashboard authentication.

### Architecture

```
Browser ──> GET /v1/auth/oidc/login ──> Redirect to IdP
Browser <── IdP authorization endpoint <──
Browser ──> GET /v1/auth/oidc/callback?code=... ──> Code exchange + ID token verification
Browser <── Set session cookie (signed JWT) <──
Browser ──> GET /dashboard/* + session cookie ──> Auth middleware validates cookie
API Client ──> Bearer: aegis_... ──> Existing API key auth (unchanged)
```

### Key Design Points

1. **Dual auth paths**: API key auth and OIDC session auth coexist. The `setupAuth()` middleware checks for a valid session cookie first, then falls back to bearer token validation.
2. **Short-lived session cookies**: After successful OIDC callback, Aegis issues a signed JWT session cookie (15-minute TTL, rotating). The JWT contains the user's `sub`, `email`, and role mapping.
3. **Role mapping**: OIDC users are assigned roles based on email domain matching (configurable). No per-user role stored in Aegis — roles come from the IdP.
4. **No user database**: Aegis remains stateless for OIDC. No user table or session store. The session cookie is self-contained (signed JWT with server secret).
5. **Configuration via env vars**: `AEGIS_OIDC_ISSUER`, `AEGIS_OIDC_CLIENT_ID`, `AEGIS_OIDC_CLIENT_SECRET`. When unset, OIDC is disabled and existing behavior is preserved.

## Alternatives Considered

### `jose` (direct implementation)
Would add only ~25KB gzipped but requires implementing 100-200 lines of security-critical OIDC protocol code (nonce verification, key rotation, token validation). Rejected due to the risk of subtle security bugs in hand-rolled OIDC.

### `@fastify/oauth2`
Provides convenient OAuth2 flow but lacks OIDC ID token validation. Would still need `jose` for token verification, resulting in two dependencies with less coverage than `openid-client` alone. Rejected.

### Passport.js
Heavy framework with many strategies. Overkill for a single IdP integration. Adds significant dependency surface. Rejected.

## Consequences

### Positive
- Enterprise-ready dashboard auth with minimal config (3 env vars)
- Full OIDC spec compliance (PKCE, nonce, key rotation) via `openid-client`
- API key auth for M2M remains completely untouched
- OIDC disabled by default — zero impact on existing deployments

### Negative
- Adds `openid-client` (~500KB unpacked) and transitive `jose` dependency
- Requires HTTPS in production (standard for OIDC)
- Session cookie management adds complexity to the auth middleware

### Risks
- Cookie signing secret must be managed securely (auto-generated if not provided)
- OIDC callback URL must be registered with the IdP

## Configuration

```bash
AEGIS_OIDC_ISSUER=https://your-org.okta.com/.well-known/oauth-authorization-server
AEGIS_OIDC_CLIENT_ID=aegis-dashboard
AEGIS_OIDC_CLIENT_SECRET=<secret>
AEGIS_OIDC_SCOPES=openid email profile
AEGIS_OIDC_COOKIE_SECRET=<auto-generated if unset>
AEGIS_OIDC_ROLE_MAP={"@company.com":"admin","*":"viewer"}
```
