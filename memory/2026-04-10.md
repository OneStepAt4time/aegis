# 2026-04-10 — Sprint Analysis, Vision, and VIBE_CODER_BIBLE

## Aegis Vision Document — 2026-04-10

### 1. La Visione: Da "Bridge" a "Middleware di Orchestrazione Enterprise"

Aegis nasce per risolvere un limite attuale di Claude Code: l'impossibilità di essere pilotato in modalità *headless* via API. Attraverso l'uso di `tmux` e del parsing del terminale, Aegis "ingabbia" Claude Code esponendolo ad agenti esterni (via REST e MCP).

Tuttavia, **il vero valore di Aegis non è l'hack di `tmux`**. Il rischio di piattaforma ("Platform Risk") è alto: Anthropic rilascerà prima o poi un'API ufficiale per Claude Code. 

La sopravvivenza e il trionfo di Aegis risiedono nel suo **Control Plane**:
* **Gestione degli Eventi:** Integrazione nativa bidirezionale con Telegram, Slack, Webhooks e SSE.
* **Sicurezza Logica e RBAC:** Intercettazione dei permessi (`permission-guard.ts`), profili utente e code di approvazione umane asincrone.
* **Standardizzazione:** Esposizione tramite Server MCP standard, rendendo Aegis un hub universale per connettere LLM e strumenti.

Quando Anthropic rilascerà l'API ufficiale, Aegis dovrà semplicemente astrarre il suo "Runner" (passando da `TmuxRunner` a `AnthropicApiRunner`). Tutta l'infrastruttura enterprise (webhook, dashboard, audit log) rimarrà il core business del prodotto.

### 2. Il Posizionamento di Mercato (Aegis vs Oh-My-Claudecode)

È fondamentale capire cosa Aegis **non** è, confrontandolo con tool popolari come Oh-My-Claudecode (OMC).

* **Oh-My-Claudecode (Il Power-Tool):** Progettato per lo sviluppatore umano davanti al terminale. Aggiunge comandi (`/team`, `/autopilot`) per estendere le funzionalità native. Soffre di *feature creep* e instabilità a causa dell'accavallamento cognitivo con Claude.
* **Aegis (Il Middleware):** Progettato per l'automazione. Si posiziona "dietro le quinte". Segue la filosofia *Separation of Concerns*: mantiene Aegis "stupido ma potente" (gestione flussi, sicurezza, log), lasciando l'intelligenza e il planning ad orchestratori esterni (es. OpenClaw) tramite API/MCP.

**Regola d'oro:** Aegis non deve diventare un framework per agenti. Niente plugin interni o esecuzione di codice cognitivo custom. L'espansione deve avvenire tramite **Server MCP** esterni e **Blueprints** (template JSON di configurazione).

### 3. Il Modello di Sicurezza (Shared Responsibility Model)

Aegis soffre del problema del *"Confused Deputy"*. Poiché l'IA agisce con i privilegi del processo Node.js host, controlli puramente software (TypeScript) non possono impedire accessi ostili al disco o alla rete se l'IA bypassa i limiti della CLI.

La soluzione è dividere la sicurezza in due strati:

1. **Livello Aegis (Sicurezza Logica):** 
 * Fornisce Audit Logging.
 * Gestisce il Controllo Umano (RBAC) via API/Telegram (`permissionMode`).
 * Garantisce che task sensibili non proseguano senza approvazione esplicita.

2. **Livello Infrastruttura (Sicurezza Fisica) - A carico dell'utente/deployment:**
 * Sandboxing tramite **Docker/Firecracker**.
 * **Egress Filtering:** Rete bloccata tranne che verso Anthropic/GitHub.
 * **Volume Mounts:** Isolamento rigoroso alla sola directory del progetto (`workDir`).

### 4. Gli Scenari d'Uso (I Tiers)

Per comunicare efficacemente questo modello, l'offerta di Aegis si divide in 3 scenari chiari:

**Scenario 1: Local Orchestration (Il Developer)**
* **Target:** Sviluppatore indipendente su Mac/Windows.
* **Sicurezza:** Implicit Trust. Aegis gira nativamente sull'OS.
* **Uso:** Lanciare task multipli in background, monitorati dalla Dashboard e approvati via Telegram.

**Scenario 2: CI/CD & Automation (Il Team)**
* **Target:** Team di sviluppo e pipeline GitHub Actions.
* **Sicurezza:** Policy-Based (Media). Fiducia nel codice, ma controlli rigorosi sulle azioni dell'IA.
* **Uso:** Uso dei *Blueprints* di Aegis (es. PR Reviewer in `permissionMode: 'plan'`). Notifiche e approvazioni centralizzate su Slack.

**Scenario 3: Zero-Trust Autonomous AI (L'Enterprise)**
* **Target:** Grandi aziende, banche, piattaforme SaaS.
* **Sicurezza:** Massima (Isolamento totale).
* **Uso:** Aegis e Claude Code girano all'interno di container Docker effimeri e isolati dalla rete aziendale. Il manager interagisce con l'API di Aegis per orchestrare i job in sicurezza.

### 5. Evoluzione Architetturale (Manager vs Worker)

Attualmente, Aegis è un **Monolite**: il server HTTP e `tmux` devono risiedere sulla stessa macchina. Questo rende inefficiente lo scaling estremo (es. 50 sessioni in 50 container separati).

**Il Futuro (Verso la v2.0): Architettura Distribuita**
Per scalare a livello Enterprise, Aegis dovrà separare il Control Plane dal Data Plane:
1. **Aegis Manager:** Il server centrale (API, Webhook, Dashboard). Non lancia mai `tmux`.
2. **Aegis Worker (Agent Node):** Container leggeri (con `tmux` e `claude`) che eseguono un demone Node.js minimo. Si connettono al Manager via WebSocket (o gRPC) per ricevere comandi ed emettere output.

*Per la versione 1.0, l'architettura monolitica è accettabile e sufficiente per gli Scenari 1 e 2. Lo Scenario 3 viene gestito istanziando un container completo per ogni task.*

### 6. Consolidamento Tecnico (Il Piano d'Azione a breve termine)

L'analisi del debito tecnico ha fatto emergere delle priorità per garantire la scalabilità e la stabilità del progetto prima di rilasciare una v1.0.

**6.1 Kill your darlings (Meno è meglio)**
Rimuovere componenti sperimentali ("aspirazionali") che inquinano la visione la codebase, come:
* **Consensus Review:** Sistema abbozzato per la revisione multi-agente, non in linea con l'essere un puro middleware.
* **Model Router:** Se non giustificato da un caso d'uso solido immediato, va eliminato o documentato chiaramente come disabilitato.
* *Regola:* Qualsiasi feature che sembra appartenere a un Agent Framework va rimossa. L'intelligenza sta fuori, Aegis è il middleware.

**6.2 Smantellare il God Object (`server.ts`)**
Con quasi 2500 righe, `server.ts` è un collo di bottiglia per la manutenzione.
* **Azione:** Usare i plugin di Fastify per estrarre la logica di business in route separate (es. `routes/sessions.ts`, `routes/pipelines.ts`, `routes/memory.ts`).
* **Vincolo:** Durante il refactoring NON modificare le firme delle API né la logica di funzionamento. I test devono continuare a passare senza modifiche.

**6.3 Realtà della Coverage (Testare i pilastri)**
Attualmente, file critici come `server.ts`, `session.ts` e `tmux.ts` (oltre 5000 righe totali) sono esclusi dal calcolo della coverage.
* **Azione:** Rimuovere l'esclusione di questi file in `vitest.config.ts` per ottenere la *vera* coverage di sistema.
* Scrivere test di integrazione mirati per le API core, invece di puntare a metriche fittizie.

**6.4 Gestione Git**
* **Azione:** Fermare i nuovi sviluppi su `develop` (che si è discostato troppo da `main`), stabilizzare e preparare un merge su `main` (es. tramite una release candidate `v0.8-rc`). Allineare i branch è prioritario per non rendere impossibile il merge futuro.
